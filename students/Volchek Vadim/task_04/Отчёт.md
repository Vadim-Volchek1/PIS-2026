## Цель работы

Реализовать прикладной слой с разделением операций на команды (изменяют состояние) и запросы (читают данные) по паттерну CQRS.

## Вариант №11 - Баг-трекер «Не баг, а фича?»

**Питч:** Отличаем баги от магии.  
**Ядро домена:** Проекты, Баги, Приоритеты, Статусы, Назначение, Вложения.

## Ход выполнения работы

### 1. Команды (Commands)

1. **CreateRequestCommand** - создание бага  
   Файл: `src/application/command/create_request_command.py`  
   Поля: `reporter_id`, `project_id`, `title`, `description`, `priority`.

2. **AssignGroupToRequestCommand** - назначение исполнителя  
   Файл: `src/application/command/assign_group_command.py`  
   Поля: `request_id`, `assignee_id`.

---

### 2. Command Handlers

**CreateRequestHandler** (`src/application/command/handlers/create_request_handler.py`):
- валидирует данные команды;
- формирует write-model бага;
- сохраняет в репозиторий;
- публикует событие `BugCreated` (если передан event publisher);
- возвращает `bug_id`.

---

### 3. Queries

1. **GetRequestByIdQuery** (`src/application/query/get_request_by_id_query.py`)

2. **RequestDto** (`src/application/query/dto/request_dto.py`)  
Read-модель для ответа на запрос чтения.

---

### 4. Query Handlers

**GetRequestByIdHandler** (`src/application/query/handlers/get_request_by_id_handler.py`):
- загружает запись из репозитория;
- возвращает `RequestDto`;
- выбрасывает ошибку, если баг не найден.

---

### 5. Application Service (Фасад)

**RequestServiceImpl** (`src/application/service/request_service_impl.py`) делегирует вызовы специализированным handlers:

| Метод | Тип | Возвращает |
|-------|-----|------------|
| `create_request(command)` | Command | `bug_id` |
| `get_request_by_id(query)` | Query | `RequestDto` |

---

### 6. Тестирование

В рамках лабораторной подготовлены сценарии тестирования handlers:

- создание бага и сохранение в repository;
- публикация события после команды;
- ошибка чтения при отсутствии записи;
- корректный маппинг read-модели `RequestDto`.

## Контрольные вопросы

1. **В чём разница между Command и Query?**  
   Command меняет состояние системы, Query только читает данные и не имеет побочных эффектов.

2. **Почему Command Handler возвращает только ID, а не весь объект?**  
   Это уменьшает связанность write/read моделей и соответствует CQRS: чтение выполняется отдельным Query.

3. **Где должна выполняться валидация?**  
   Базовая валидация формата и пустых полей - в Command DTO, доменные инварианты - в доменной модели/агрегате.

4. **Можно ли вызывать Query из Command Handler?**  
   Обычно не рекомендуется, чтобы не смешивать ответственности. Лучше возвращать ID и читать данные отдельным запросом.

5. **Зачем разделять Request DTO от клиента и внутренний Command?**  
   Внешний DTO зависит от транспорта, а внутренний Command - от бизнес-кейса. Это изолирует Application Layer от API-деталей.
