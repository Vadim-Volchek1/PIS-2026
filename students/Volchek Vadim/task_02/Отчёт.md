### Часть 3. Domain Layer

**Entity:** Bug (в файле `domain/request.py`)  
Содержит `id`, `project_id`, `title`, `description`, `priority`, `status`, `assignee_id`.

**Value Object:** Priority (в файле `domain/zone.py`)  
Фиксированные значения: `LOW`, `MEDIUM`, `HIGH`, `CRITICAL`.

**Enum статусов:** BugStatus (в файле `domain/request_status.py`)  
Жизненный цикл: `OPEN -> IN_PROGRESS -> RESOLVED -> CLOSED`.

**Бизнес-правила:**
1. Заголовок бага обязателен.
2. Нельзя закрыть баг, если он не в статусе `RESOLVED`.
3. Баг может иметь только одного активного исполнителя.

---

### Часть 4. Application Layer

#### Входящий порт

`CreateBugUseCase` (в файле `application/port/in/create_request_use_case.py`):
- принимает `CreateBugCommand`;
- возвращает `bug_id`.

#### Исходящие порты

`BugRepository` (в файле `application/port/out/request_repository.py`):
- `save()`, `find_by_id()`, `find_by_status()`.

`NotificationService` (в файле `application/port/out/notification_service.py`):
- `send_assigned_notification()`.

#### Application Service

`BugService` (в файле `application/service/request_service.py`):
1. Создаёт доменный объект бага.
2. Сохраняет в репозитории.
3. Назначает исполнителя.
4. Отправляет уведомление.
5. Возвращает `bug_id`.

---

### Часть 5. Infrastructure Layer

#### Входящий адаптер: REST API

`BugController` (`infrastructure/adapter/in/request_controller.py`)
- `POST /api/bugs` - создание бага
- `GET /api/health` - проверка сервиса

#### Исходящий адаптер: Repository

`InMemoryBugRepository` (`infrastructure/adapter/out/in_memory_request_repository.py`)
- хранение багов в памяти (`dict`).

#### Исходящий адаптер: Notification

`MockNotificationService` (`infrastructure/adapter/out/mock_sms_service.py`)
- имитация отправки уведомления в консоль.

---

### Часть 6. Dependency Injection

`DependencyContainer` (`infrastructure/config/dependency_injection.py`):
- создаёт адаптеры;
- связывает их с `BugService`;
- конфигурирует веб-приложение.
