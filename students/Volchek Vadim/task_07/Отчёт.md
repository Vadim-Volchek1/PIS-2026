## Цель работы

Реализовать CQRS с разделением Write Model и Read Model для баг-трекера.

---

## Вариант №11 - Баг-трекер «Не баг, а фича?»

---

## Ход выполнения работы

### 1. Write Model

**Агрегат:** `Bug` (используется как `Request` в учебном шаблоне).

**Структура:**
- Нормализованные таблицы: `bugs`, `projects`, `bug_assignees`.
- Проверка инвариантов в доменной модели (статусы, допустимые переходы, обязательность приоритета).

---

### 2. Read Model

**Проекция:** `RequestView` (денормализованное представление бага для чтения).

**Структура:**
- Денормализованная таблица/материализованное представление `bug_view`.
- Предзагруженные поля: проект, исполнитель, приоритет, вычисляемая длительность.

---

### 3. Event-Driven Sync

**События:**
- `BugCreated` -> создать запись в `bug_view`.
- `BugAssigned` -> обновить `assignee_id` и статус.
- `BugActivated` -> обновить поле `activated_at`.
- `BugClosed` -> обновить `completed_at` и `duration_minutes`.

**Код обработчика проекций:** `src/cqrs/projection/request_projection.py`

## Вывод

Реализовано разделение командной и запросной модели для баг-трекера: операции изменения состояния выполняются через Write Model, а чтение для UI обеспечивается через денормализованную Read Model.  
Синхронизация моделей организована событийным способом, что повышает производительность чтения и сохраняет масштабируемость архитектуры.
