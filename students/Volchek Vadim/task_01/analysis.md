# Анализ границ ответственности

**Система:** Баг-трекер «Не баг, а фича?» 🐛  
**Сценарий:** Создание бага и назначение исполнителя

---

## 1. Транзакционные границы

### 1.1. Где начинается и заканчивается транзакция?

- **Начало:** пользователь нажимает `Создать баг`.
- **Конец:** запись бага и назначение исполнителя подтверждены в БД.

Выделены 2 транзакции:
1. **Tx1:** создание карточки бага + сохранение вложений + приоритет.
2. **Tx2:** выбор исполнителя + фиксация назначения + смена статуса на `IN_PROGRESS`.

Если `Tx2` не прошла, баг остаётся в `OPEN` и требует ручного назначения.

### 1.2. Какие операции атомарные?

**Tx1 (создание бага):**
1. Валидация обязательных полей.
2. `INSERT` в `bugs`.
3. `INSERT` вложений в `bug_attachments`.
4. Запись в outbox события `BugCreated`.

**Tx2 (назначение):**
1. Проверка доступности разработчика.
2. `INSERT` в `bug_assignments`.
3. `UPDATE bugs SET status='IN_PROGRESS'`.
4. Запись в outbox события `BugAssigned`.

### 1.3. Какие операции асинхронные?

| Операция | Причина |
|----------|---------|
| Отправка email/slack уведомлений | Не влияет на инварианты бага |
| Обновление read-модели дашборда | Допустима eventual consistency |
| Агрегация статистики по приоритетам | Может обновляться фоново |

---

## 2. Таблица транзакционных границ

| Операция | Синхронная/Асинхронная | Откат при ошибке | Retry-стратегия | Идемпотентность |
|----------|------------------------|------------------|-----------------|-----------------|
| Валидация полей бага | Синхронная | Нет | Нет | Да |
| Создание записи в `bugs` | Синхронная | Да (ROLLBACK Tx1) | Нет | Да (`idempotency_key`) |
| Сохранение вложений | Синхронная | Да (ROLLBACK Tx1) | Нет | Да (hash файла) |
| Публикация `BugCreated` через outbox | Синхронная (в Tx1) | Да | Повтор чтением outbox | Да (`event_id`) |
| Проверка доступности разработчика | Синхронная | Да (ROLLBACK Tx2) | Нет | Да |
| Назначение исполнителя | Синхронная | Да (ROLLBACK Tx2) | Нет | Да (unique `bug_id`) |
| Публикация `BugAssigned` | Синхронная (в Tx2) | Да | Повтор чтением outbox | Да |
| Email/Slack уведомление | Асинхронная | Нет | 5 попыток, backoff 1-2-4-8-16s | Да |

## 3. Обработка исключительных ситуаций

### 3.1. База данных недоступна

- **Обнаружение:** `DatabaseTimeout`/`ConnectionRefused`.
- **Реакция:** полный rollback активной транзакции.
- **Компенсация:** не требуется, так как запись не подтверждена.
- **Уведомление:** `Не удалось сохранить баг. Повторите попытку позже`.

### 3.2. Нет доступных исполнителей

- **Обнаружение:** запрос в `users` вернул пустой набор.
- **Реакция:** баг остаётся в `OPEN`, создаётся событие `BugNeedsAssignment`.
- **Компенсация:** не требуется.
- **Уведомление:** менеджеру отправляется задача на ручное назначение.

### 3.3. Notification Service недоступен

- **Обнаружение:** timeout внешнего API.
- **Реакция:** баг не откатывается, уведомления отправляются позже через очередь.
- **Компенсация:** повторная отправка с ограничением числа попыток.
- **Уведомление:** `Баг создан, уведомление исполнителю будет отправлено позже`.

---

## 4. Идемпотентность

Для endpoint `POST /api/bugs` используется `idempotency_key`:

1. При первом запросе создаётся баг и сохраняется ответ.
2. При повторном запросе с тем же ключом возвращается тот же `bug_id`.
3. TTL ключа - 24 часа.

Это защищает от дублей при повторном клике и сетевых повторах.

---

## 5. Вывод

- Транзакции разделены на создание бага и назначение исполнителя.
- Критичные операции выполняются синхронно и атомарно.
- Внешние интеграции (уведомления) вынесены в асинхронный контур.
- Принцип: сначала консистентность данных в БД, затем побочные эффекты.
