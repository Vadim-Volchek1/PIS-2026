### 2. Use-case описание

👉 **Ссылка на файл:** [../use-case.md](../use-case.md)

**Основной сценарий:** Создание бага и назначение исполнителя.  
**Первичный актор:** Тестировщик.  
**Цель:** Зафиксировать дефект, назначить ответственного разработчика, уведомить участников.

**Краткое описание основного потока:**
1. Тестировщик заполняет карточку бага.
2. Система валидирует данные и сохраняет баг.
3. Система назначает доступного разработчика.
4. Публикуются доменные события и отправляются уведомления.

**Альтернативные потоки:** сохранение черновика, создание бага без автоназначения.  
**Исключительные ситуации:** ошибка валидации вложения, недоступность БД, недоступность Notification Service.

#### 3.1. Happy Path (успешный сценарий)

👉 **PlantUML исходник:** [../src/diagrams/sequence-happy.puml](../src/diagrams/sequence-happy.puml)

### 5. Анализ границ ответственности

👉 **Ссылка на файл:** [../src/analysis.md](../src/analysis.md)

#### 5.1. Транзакционные границы

| Операция | Синхронная/Асинхронная | Откат при ошибке | Retry-стратегия | Идемпотентность |
|----------|------------------------|------------------|-----------------|-----------------|
| Создание записи бага | Синхронная | Да (Tx1 rollback) | Нет | Да |
| Сохранение вложений | Синхронная | Да (Tx1 rollback) | Нет | Да |
| Назначение исполнителя | Синхронная | Да (Tx2 rollback) | Нет | Да |
| Отправка уведомлений | Асинхронная | Нет | 5 попыток с backoff | Да |

#### 5.2. Обработка исключительных ситуаций

**Реализовано стратегий обработки:** 3

##### Исключительная ситуация 1: Недоступность БД

- **Условие возникновения:** таймаут подключения.
- **Обнаружение:** исключение `DatabaseTimeout`.
- **Реакция:** rollback транзакции.
- **Компенсация:** не требуется.
- **Уведомление пользователя:** "Не удалось сохранить баг. Повторите попытку позже."

##### Исключительная ситуация 2: Нет доступного разработчика

- **Условие возникновения:** выборка `AVAILABLE` разработчиков пуста.
- **Обнаружение:** пустой результат запроса.
- **Реакция:** баг остаётся в статусе `OPEN`.
- **Компенсация:** не требуется.
- **Уведомление пользователя:** "Баг создан без назначения, требуется ручное распределение."

##### Исключительная ситуация 3: Недоступен сервис уведомлений

- **Условие возникновения:** timeout внешнего API.
- **Обнаружение:** ошибка в Notification Worker.
- **Реакция:** запись в retry очередь.
- **Компенсация:** повторная отправка.
- **Уведомление пользователя:** "Баг создан, уведомления будут отправлены позже."

## Контрольные вопросы

1. Что такое транзакционная граница? Где она проходит в вашем сценарии?  
   - Граница проходит между сохранением бага и назначением исполнителя: это две независимые транзакции.
2. Почему операция назначения синхронная, а уведомление асинхронное?  
   - Назначение влияет на консистентность данных, уведомление - побочный эффект.
3. Как обеспечить идемпотентность?  
   - Использовать `idempotency_key` и возвращать сохранённый ответ при повторном запросе.
4. Что произойдёт при ошибке внешнего сервиса после назначения?  
   - Назначение не откатывается, уведомление уходит в retry очередь.
5. Как система обнаружит недоступность внешнего сервиса?  
   - По timeout/HTTP ошибке при вызове API уведомлений.
6. Какие данные нужно логировать?  
   - `bug_id`, `project_id`, `assignee`, код ошибки, timestamp, retry_count.
