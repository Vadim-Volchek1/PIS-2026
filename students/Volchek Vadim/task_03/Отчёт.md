Научиться применять тактические паттерны DDD (Entities, Value Objects, Aggregates, Domain Events) для реализации доменного слоя с инвариантами и бизнес-логикой.

## Ход выполнения работы

### 1. Value Objects

1. **Zone** (`src/domain/models/zone.py`)  
   Используется как Value Object приоритета бага (`LOW`, `MEDIUM`, `HIGH`, `CRITICAL`) и SLA-диапазона.

2. **PhoneNumber** (`src/domain/models/phone_number.py`)  
   Иммутабельный контакт ответственного с валидацией формата.

3. **RequestStatus** (`src/domain/models/request_status.py`)  
   Enum статусов бага: `DRAFT -> ACTIVE -> COMPLETED` с возможностью отмены.

---

### 2. Entities

1. **Group** (`src/domain/models/group.py`)  
   Команда разработчиков, назначаемая на баг.

2. **Request** (`src/domain/models/request.py`)  
   Сущность заявленного бага (используется как Aggregate Root).

---

### 3. Aggregate Root

**Aggregate Root:** `Request`

**Границы агрегата:**
- Корень: `Request`
- Внутренние сущности: `Group`
- Value Objects: `Zone`, `RequestStatus`

**Инварианты агрегата:**

| № | Инвариант | Как проверяется |
|---|----------|----------------|
| 1 | Нельзя активировать баг без команды | `activate()` |
| 2 | Назначение команды только для `DRAFT` | `assign_group()` |
| 3 | Нельзя менять приоритет у закрытого бага | `change_zone()` |

---

### 4. Domain Events

События реализованы в `src/domain/events/request_events.py`:

- `GroupAssignedToRequest`
- `RequestActivated`
- `RequestZoneChanged`
- `RequestCompleted`

События регистрируются в агрегате `Request` при каждом значимом изменении состояния.

---

### 5. Юнит-тесты

Тесты: `src/tests/test_request.py`

Проверяются:
- запрет активации без команды;
- регистрация событий при назначении и активации;
- запрет смены приоритета у закрытого бага.

## Контрольные вопросы

1. **В чём отличие Value Object от Entity?**  
   Value Object сравнивается по значению и обычно иммутабелен, а Entity имеет устойчивую идентичность (ID) и сравнивается по ней.

2. **Почему Aggregate Root должен инкапсулировать доступ к внутренним сущностям?**  
   Чтобы централизованно защищать инварианты и не допускать обхода бизнес-правил через прямые изменения внутренних объектов.

3. **Какая роль Domain Events? Приведите пример из вашей системы.**  
   Domain Events фиксируют важные доменные изменения и позволяют реактивно выполнять побочные процессы. Пример: `RequestActivated` при переводе бага в работу.

4. **Как вы проверяете инварианты в вашем агрегате?**  
   Инварианты проверяются в методах агрегата (`assign_group`, `activate`, `change_zone`, `complete`) и при нарушении выбрасывается `ValueError`.

5. **Почему Value Objects делаются иммутабельными?**  
   Это исключает скрытые побочные эффекты и делает модель предсказуемой: объект после создания всегда валиден и не меняется.
