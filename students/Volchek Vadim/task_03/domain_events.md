# Доменные события в `task_03`

## Зачем нужны domain events
Доменные события фиксируют факт уже совершившегося бизнес-изменения в агрегате. Это позволяет:
- построить аудит изменения багов;
- асинхронно уведомлять смежные контексты;
- обновлять read-model без связности на уровне доменного слоя.

## Реализованные события

### 1) `BugCreated`
- **Файл:** `src/domain/events/bug_created.py`
- **Когда возникает:** сразу после создания экземпляра `Bug`.
- **Поля:** `bug_id`, `project_id`, `priority`, `occurred_at`.

### 2) `BugAssigned`
- **Файл:** `src/domain/events/bug_assigned.py`
- **Когда возникает:** после успешного назначения исполнителя через `assign_to()`.
- **Поля:** `bug_id`, `assignee_id`, `occurred_at`.

### 3) `BugStatusChanged`
- **Файл:** `src/domain/events/bug_status_changed.py`
- **Когда возникает:** после валидной смены статуса через `change_status()`.
- **Поля:** `bug_id`, `old_status`, `new_status`, `occurred_at`.

## Где события регистрируются

В `src/domain/models/bug.py` события добавляются в внутренний буфер `_events`, затем извлекаются методом `pull_domain_events()`.

```python
def pull_domain_events(self) -> List[object]:
    events = list(self._events)
    self._events.clear()
    return events
```

## Гарантии и ограничения

- Доменный слой только генерирует события, но не отправляет их в брокер напрямую.
- Публикация наружу выполняется на application/infrastructure уровнях.
- События иммутабельны (`@dataclass(frozen=True)`), что исключает изменение уже зафиксированного факта.

## Проверка в тестах
Файл `src/tests/test_bug_invariants.py`, сценарий `test_domain_events_are_collected_and_cleared` подтверждает:
1. после операций внутри агрегата события накапливаются;
2. после `pull_domain_events()` буфер очищается.

## Вывод
Выделение доменных событий формирует устойчивую основу для CQRS и интеграций в следующих лабораторных работах, сохраняя чистоту и изоляцию доменной модели.
